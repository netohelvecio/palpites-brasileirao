# Plan 6.5 — Oracle VM Provisioning + Primeiro Deploy Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** VM Oracle ARM Ampere provisionada, hardenizada e rodando a stack de produção, com TLS automático via Let's Encrypt e WhatsApp pareado.

**Architecture:** Runbook operacional (não código de produto) que executa: criação da VM no console Oracle → hardening (ufw, fail2ban, user deploy, SSH keys) → instalação Docker + OCI CLI → clone do repo em `/opt/palpites` → criação de `.env.production` + `db_password.txt` → `docker compose up` → pareamento Baileys → smoke test público via DuckDNS.

**Tech Stack:** Console Oracle Cloud, Ubuntu 22.04 ARM, ufw, fail2ban, Docker, OCI CLI, DuckDNS, Caddy, Baileys.

**Spec:** `docs/superpowers/specs/2026-04-27-deploy-design.md`

**Depende de:**
- Plan 6.1 (Dockerfile, compose.prod.yml, Caddyfile commitados)
- Plan 6.2 (CI verde — confirma que código compila e testes passam antes de subir)
- Plan 6.3 (deploy.yml commitado — mas **não merged em main** ainda)
- Plan 6.4 (backup.sh + runbooks docs commitados)

### Task 1: Criar VM no console Oracle

**Files:** (operação no console web)

- [ ] **Step 1: Login e navegação**

`cloud.oracle.com` → login → Hambúrguer (canto superior esquerdo) → **Compute → Instances → Create Instance**.

- [ ] **Step 2: Configurar a instância**

Campos:

| Campo | Valor |
|---|---|
| Name | `palpites-prod` |
| Compartment | (default) |
| Placement → Availability domain | qualquer (testar AD-1, depois AD-2/AD-3 se "out of capacity") |
| Image | **Edit → Change image → Ubuntu 22.04** (marca verde "Always Free Eligible") |
| Shape | **Edit → Change shape → Ampere → VM.Standard.A1.Flex** com **1 OCPU + 6 GB RAM** |
| Networking | Create new VCN; "Assign a public IPv4 address" marcado |
| SSH keys | "Paste public keys" → conteúdo de `~/.ssh/id_ed25519.pub` (sua chave local) |
| Boot volume | 50 GB (default — não mexer) |

**Create**.

Se "out of capacity" (erro vermelho na criação): trocar AD ou tentar de novo a cada 2–3 min. Capacidade ARM no Vinhedo é limitada mas abre.

- [ ] **Step 3: Anotar IP público**

Aguardar `RUNNING` (~1 min). Anotar **Public IP** no campo da página da instância.

### Task 3: Hardening da VM (primeiro acesso)

**Files:** (operação por SSH)

- [ ] **Step 1: SSH como `ubuntu` (user padrão Oracle)**

```bash
ssh ubuntu@<IP-PUBLICO>
```

- [ ] **Step 2: Update e ferramentas básicas**

```bash
sudo apt update && sudo apt upgrade -y
sudo apt install -y ufw fail2ban unattended-upgrades curl ca-certificates gnupg
```

- [ ] **Step 3: Flush iptables (PEGADINHA Oracle)**

```bash
sudo iptables -F
sudo iptables -X
sudo iptables -P INPUT ACCEPT
sudo iptables -P FORWARD ACCEPT
sudo iptables -P OUTPUT ACCEPT
sudo netfilter-persistent save
```

- [ ] **Step 4: Configurar ufw como firewall único**

```bash
sudo ufw default deny incoming
sudo ufw default allow outgoing
sudo ufw allow 22/tcp
sudo ufw allow 80/tcp
sudo ufw allow 443/tcp
sudo ufw --force enable
sudo ufw status verbose
```

Expected: status `active`, regras `22, 80, 443 ALLOW IN`.

- [ ] **Step 5: Criar usuário deploy**

```bash
sudo adduser --disabled-password --gecos "" deploy
sudo usermod -aG sudo deploy
sudo mkdir -p /home/deploy/.ssh
sudo cp ~/.ssh/authorized_keys /home/deploy/.ssh/
sudo chown -R deploy:deploy /home/deploy/.ssh
sudo chmod 700 /home/deploy/.ssh
sudo chmod 600 /home/deploy/.ssh/authorized_keys
```

- [ ] **Step 6: Adicionar chave do GitHub Actions (do Plan 6.3) ao `authorized_keys` do deploy**

Localmente (na sua máquina), gerar par dedicado se ainda não tiver:

```bash
ssh-keygen -t ed25519 -f ~/.ssh/palpites_deploy -C "github-actions-deploy" -N ""
```

Copiar `.pub` pra VM:

```bash
cat ~/.ssh/palpites_deploy.pub | ssh ubuntu@<IP-PUBLICO> \
  "sudo tee -a /home/deploy/.ssh/authorized_keys"
```

(Note: usando `ubuntu` ainda; testar acesso `deploy` no próximo step.)

- [ ] **Step 7: Validar SSH como deploy**

Da sua máquina:

```bash
ssh -i ~/.ssh/palpites_deploy deploy@<IP-PUBLICO> "echo ok"
```

Expected: `ok`.

- [ ] **Step 8: SSH só com chave (já default no Oracle, mas garantir)**

De volta na VM como `ubuntu`:

```bash
sudo sed -i 's/^#\?PasswordAuthentication.*/PasswordAuthentication no/' /etc/ssh/sshd_config
sudo systemctl restart ssh
```

- [ ] **Step 9: Auto-updates de segurança**

```bash
sudo dpkg-reconfigure -plow unattended-upgrades   # selecionar "yes"
```

### Task 5: Clonar repo e configurar segredos

**Files:** (operação por SSH)

- [ ] **Step 1: Diretório**

```bash
sudo mkdir -p /opt/palpites
sudo chown deploy:deploy /opt/palpites
cd /opt
git clone https://github.com/<usuario>/palpites-brasileirao.git palpites
cd palpites
git checkout main
```

(Branch `main` não tem o `deploy.yml` ainda — só será merged depois de tudo estar configurado.)

- [ ] **Step 2: Substituir CHANGE_ME no compose pelo username GitHub real**

```bash
sed -i "s|ghcr.io/CHANGE_ME/palpites-api|ghcr.io/<usuario-lowercase>/palpites-api|g" \
  docker-compose.prod.yml
```

(Esse `sed` é local na VM e **não vai pro repo**. Documentado aqui pra você lembrar.)

- [ ] **Step 3: `.env.production`**

```bash
cp apps/api/.env.production.example apps/api/.env.production
nano apps/api/.env.production
```

Preencher cada variável:
- `APP_KEY=$(openssl rand -hex 32)` — gerar e colar
- `ADMIN_API_TOKEN=$(openssl rand -hex 32)` — gerar e colar; **guardar este valor** (precisa pra chamar a API admin depois)
- `FOOTBALL_DATA_TOKEN=` — token gratuito de https://www.football-data.org/client/register
- `WHATSAPP_GROUP_JID=` — deixar vazio por agora (preenche após pareamento)
- demais valores conforme `.env.production.example`

```bash
chmod 600 apps/api/.env.production
```

- [ ] **Step 4: Senha do Postgres (Docker secret)**

```bash
mkdir -p infra/secrets
openssl rand -base64 32 > infra/secrets/db_password.txt
chmod 600 infra/secrets/db_password.txt
```

Copiar o conteúdo pra `DB_PASSWORD` no `.env.production`:

```bash
DB_PW=$(cat infra/secrets/db_password.txt)
sed -i "s|^DB_PASSWORD=.*$|DB_PASSWORD=${DB_PW}|" apps/api/.env.production
```

### Task 6: Login no GHCR e primeiro `docker compose up`

**Files:** (operação por SSH)

- [ ] **Step 1: Personal Access Token do GitHub**

Localmente: GitHub → Settings → Developer settings → Personal access tokens → **Tokens (classic)** → Generate. Scope: **`read:packages`**. Copiar token.

- [ ] **Step 2: `docker login` na VM**

```bash
echo "<TOKEN>" | docker login ghcr.io -u <usuario> --password-stdin
```

Expected: `Login Succeeded`.

- [ ] **Step 3: Bootstrap da imagem no GHCR**

A imagem `:latest` não existe no GHCR ainda (deploy.yml não foi merged em `main`). Duas opções equivalentes pra publicar a primeira versão:

**Opção A — workflow_dispatch no GitHub** (mais simples se você já mergeou Plan 6.3 em alguma branch):

GitHub repo → Actions → "deploy" workflow → **Run workflow** → selecionar branch da feature 6.3.

O job `build` roda e empurra `:latest` + `:<sha>`. O job `deploy` é pulado pelo `if: github.ref == 'refs/heads/main'` adicionado no plano 6.3.

**Opção B — build local com buildx** (não depende do GHA):

Localmente (não na VM):

```bash
docker buildx create --use --name palpites-builder
docker login ghcr.io -u <usuario>      # PAT com write:packages
docker buildx build \
  --platform linux/arm64 \
  -f apps/api/Dockerfile \
  -t ghcr.io/<usuario>/palpites-api:latest \
  --push .
```

Build emulado leva ~3–5 min.

- [ ] **Step 4: Subir a stack na VM**

```bash
cd /opt/palpites
docker compose -f docker-compose.prod.yml pull
docker compose -f docker-compose.prod.yml up -d
docker compose -f docker-compose.prod.yml ps
```

Expected: 3 containers `running`. Caddy pode demorar 30-60s pra emitir cert Let's Encrypt na primeira requisição HTTPS.

- [ ] **Step 5: Migrations**

```bash
docker compose -f docker-compose.prod.yml exec api node ace migration:run --force
```

Expected: `Executed N migrations`.

### Task 7: Pareamento Baileys

- [ ] **Step 1: Configurar `WHATSAPP_MODE=real` no `.env.production`**

```bash
sed -i 's|^WHATSAPP_MODE=.*$|WHATSAPP_MODE=real|' apps/api/.env.production
docker compose -f docker-compose.prod.yml restart api
```

- [ ] **Step 2: Acompanhar logs e escanear QR**

```bash
docker compose -f docker-compose.prod.yml logs -f api
```

Aguardar QR code ASCII no terminal. **Em outra janela do celular** (ou no chip dedicado):
1. WhatsApp → Configurações → Dispositivos conectados → Conectar dispositivo
2. Escanear o QR exibido no terminal SSH

Expected log: `Connection open`.

`Ctrl+C` no `logs -f` (não derruba o container).

- [ ] **Step 3: Listar grupos pra obter o JID do bolão**

```bash
docker compose -f docker-compose.prod.yml exec api node ace whatsapp:list-groups
```

Expected: lista com JIDs `xxxxx@g.us`. Identificar o do grupo do bolão e copiar.

- [ ] **Step 4: Salvar JID no `.env.production` e restart**

```bash
nano apps/api/.env.production    # WHATSAPP_GROUP_JID=<jid>@g.us
docker compose -f docker-compose.prod.yml restart api
```

- [ ] **Step 5: Validar reconnect com sessão persistida**

```bash
docker compose -f docker-compose.prod.yml logs --tail=50 api | grep -i "connection open"
```

Expected: `Connection open` apareceu sem QR — sessão restaurada do volume.

### Task 8: DuckDNS + smoke test HTTPS público

**Files:** (operação por SSH)

- [ ] **Step 1: Configurar DuckDNS**

Seguir `docs/runbooks/duckdns-setup.md` (Plan 6.4).

Resumo:
1. Criar conta + subdomínio em `duckdns.org`, copiar token
2. Apontar manualmente o subdomínio pro IP da VM no painel DuckDNS
3. Cron na VM:

```bash
TOKEN="<token>"
echo "*/5 * * * * deploy curl -s 'https://www.duckdns.org/update?domains=palpites-bolao&token=${TOKEN}&ip=' >> /var/log/duckdns.log 2>&1" \
  | sudo tee /etc/cron.d/duckdns
sudo chmod 644 /etc/cron.d/duckdns
sudo touch /var/log/duckdns.log
sudo chown deploy:deploy /var/log/duckdns.log
```

- [ ] **Step 2: Confirmar resolução DNS**

```bash
dig +short palpites-bolao.duckdns.org
```

Expected: IP da VM.

- [ ] **Step 3: Smoke test HTTPS via Caddy**

```bash
curl -i https://palpites-bolao.duckdns.org/health
```

Expected: `HTTP/2 200` + body com `{"status":"ok"}`.

Primeira requisição pode demorar 10–30s (Caddy emite cert Let's Encrypt). Tentativas seguintes são instantâneas.

- [ ] **Step 4: Smoke test API admin**

```bash
ADMIN_TOKEN="<conteúdo-do-ADMIN_API_TOKEN-do-env>"
curl -s -H "Authorization: Bearer ${ADMIN_TOKEN}" \
  https://palpites-bolao.duckdns.org/api/v1/seasons | jq
```

Expected: lista vazia ou seasons existentes (depende se o banco tem dados).
