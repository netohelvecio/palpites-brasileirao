# Plan 6.3 — GitHub Actions Deploy + GHCR Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Push em `main` → build ARM64 da imagem da API → push pra GHCR → SSH na VM Oracle → `docker compose pull` + `up -d` + `migration:run`.

**Architecture:** Workflow `.github/workflows/deploy.yml` com dois jobs sequenciais: `build` (buildx multi-arch via QEMU + push pra `ghcr.io/<owner>/palpites-api:<sha>` e `:latest`) e `deploy` (SSH via `appleboy/ssh-action` rodando comandos no `/opt/palpites` da VM). Cache do buildx usa `type=gha`.

**Tech Stack:** docker/setup-qemu-action, docker/setup-buildx-action, docker/build-push-action, appleboy/ssh-action, GHCR.

**Spec:** `docs/superpowers/specs/2026-04-27-deploy-design.md`

**Depende de:**
- Plan 6.1 (Dockerfile, compose.prod.yml, Caddyfile já versionados no repo)
- Plan 6.5 (VM provisionada, repo clonado em `/opt/palpites`, secrets do GitHub configurados)

### Task 1: Criar `.github/workflows/deploy.yml`

**Files:**
- Create: `.github/workflows/deploy.yml`

- [ ] **Step 1: Conteúdo do workflow**

```yaml
name: deploy

on:
  push:
    branches: [main]
  workflow_dispatch:

concurrency:
  group: deploy-prod
  cancel-in-progress: false

jobs:
  build:
    runs-on: ubuntu-latest
    permissions:
      contents: read
      packages: write
    outputs:
      image: ${{ steps.meta.outputs.image }}
    steps:
      - uses: actions/checkout@v4

      - uses: docker/setup-qemu-action@v3
        with:
          platforms: arm64

      - uses: docker/setup-buildx-action@v3

      - uses: docker/login-action@v3
        with:
          registry: ghcr.io
          username: ${{ github.actor }}
          password: ${{ secrets.GITHUB_TOKEN }}

      - name: Resolve image name
        id: meta
        run: |
          owner_lower=$(echo "${{ github.repository_owner }}" | tr '[:upper:]' '[:lower:]')
          echo "image=ghcr.io/${owner_lower}/palpites-api" >> "$GITHUB_OUTPUT"

      - name: Build and push
        uses: docker/build-push-action@v5
        with:
          context: .
          file: ./apps/api/Dockerfile
          platforms: linux/arm64
          tags: |
            ${{ steps.meta.outputs.image }}:${{ github.sha }}
            ${{ steps.meta.outputs.image }}:latest
          push: true
          cache-from: type=gha
          cache-to: type=gha,mode=max

  deploy:
    needs: build
    runs-on: ubuntu-latest
    environment: production
    steps:
      - name: Deploy via SSH
        uses: appleboy/ssh-action@v1
        with:
          host: ${{ secrets.DEPLOY_HOST }}
          username: ${{ secrets.DEPLOY_USER }}
          key: ${{ secrets.DEPLOY_SSH_KEY }}
          script_stop: true
          script: |
            set -euo pipefail
            cd /opt/palpites
            git fetch origin main
            git reset --hard origin/main
            docker compose -f docker-compose.prod.yml pull api
            docker compose -f docker-compose.prod.yml up -d --no-deps api
            sleep 5
            docker compose -f docker-compose.prod.yml exec -T api node ace migration:run --force
            curl --fail --silent --show-error --max-time 10 http://localhost:8080/health > /dev/null \
              || (echo "health check failed via local Caddy"; exit 1)
```

**Notas:**
- `concurrency.group: deploy-prod, cancel-in-progress: false` — garante que no máximo 1 deploy roda por vez, mas não cancela um deploy em andamento (terminar é melhor que deixar pela metade).
- `git reset --hard origin/main` em vez de `git pull` — evita conflitos se houver edição manual no servidor.
- `script_stop: true` faz o SSH parar no primeiro comando que falhar.
- O `curl http://localhost:8080/health` no fim... espera, o Caddy de produção escuta em `:80`/`:443`, não `:8080`. Esse path é do `Caddyfile.local`. Em prod precisa usar `https://palpites-bolao.duckdns.org/health` ou `http://localhost` direto contra o container `api`. Ajustar:

Substituir a linha do `curl` por:

```bash
docker compose -f docker-compose.prod.yml exec -T api curl --fail --silent --max-time 5 http://127.0.0.1:3333/health > /dev/null
```

Validar dentro do próprio container, evita depender de DNS/TLS no momento do deploy.

```markdown
# GitHub Secrets — palpites-brasileirao

Configurar em **Settings → Secrets and variables → Actions** do repo.

## Repository secrets

| Nome | Origem | Notas |
|---|---|---|
| `DEPLOY_HOST` | IP público da VM Oracle | Ex: `192.0.2.42` |
| `DEPLOY_USER` | `deploy` | Usuário não-root criado no provisioning (Plan 6.5) |
| `DEPLOY_SSH_KEY` | chave privada SSH do `deploy@VM` | OpenSSH format. Gerar com `ssh-keygen -t ed25519 -C "github-actions"` localmente, colocar `.pub` em `/home/deploy/.ssh/authorized_keys` na VM |

`GITHUB_TOKEN` é automático — não configurar manualmente.

## Environment "production"

Criar environment chamado `production` em **Settings → Environments → New environment**.

- Required reviewers (opcional): adicionar você mesmo se quiser confirmar manual antes de cada deploy.
- Não precisa de secrets adicionais — o workflow `deploy.yml` referencia `environment: production` e usa os repository secrets acima.

## Como gerar `DEPLOY_SSH_KEY`

Localmente (não na VM):

```bash
ssh-keygen -t ed25519 -f ~/.ssh/palpites_deploy -C "github-actions-deploy"
# senha vazia (Actions não suporta passphrase)
```

Resultado:
- `~/.ssh/palpites_deploy` — chave privada → conteúdo no secret `DEPLOY_SSH_KEY`
- `~/.ssh/palpites_deploy.pub` — chave pública → adicionar a `/home/deploy/.ssh/authorized_keys` na VM

```bash
cat ~/.ssh/palpites_deploy.pub | ssh deploy@<IP> "cat >> ~/.ssh/authorized_keys"
```

Testar: `ssh -i ~/.ssh/palpites_deploy deploy@<IP> "echo ok"` → `ok`.
```

### Task 4: Garantir que o pacote GHCR é público (ou autenticado)

**Files:** (nenhum, é configuração no GitHub UI)

- [ ] **Step 1: Após primeiro push da imagem**

GHCR cria pacote `palpites-api` automaticamente como **privado**. Pull autenticado funciona via `docker login ghcr.io` na VM (Plan 6.5).

Alternativamente, tornar pacote público:
1. Github → seu profile → Packages → palpites-api
2. Package settings → "Change visibility" → Public

Pra um app de bolão pessoal, **privado é o ideal** — não é segredo crítico mas evita curiosos. Documentar que a VM precisa de `docker login`.
