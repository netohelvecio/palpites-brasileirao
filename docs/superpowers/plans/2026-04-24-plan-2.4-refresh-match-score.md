# Plano 2.4 — Refresh Match Score

> **For agentic workers:** REQUIRED SUB-SKILL: superpowers:executing-plans + superpowers:test-driven-development.

**Goal:** Endpoint que consulta `/matches/{externalId}` da football-data.org e atualiza `homeScore`, `awayScore`, `status` e `kickoffAt` do `Match` local. Base pra scheduler disparar periodicamente no plano 3.

**Architecture:**
- Função pura `mapMatchStatus(FootballDataMatchStatus) → MatchStatus` em `app/services/match_status_mapper.ts`.
- `RefreshMatchService.refresh(matchId)` em `app/services/refresh_match_service.ts`:
  - `MatchRepository.findByIdOrFail`
  - `FootballDataClient.fetchMatchById(externalId)`
  - Se `null` (404 no provider): reporta `not-found`, não apaga local.
  - Caso contrário: atualiza placar, status, kickoff.
- `MatchesController.refresh` → `POST /api/v1/matches/:id/refresh`.

Edge cases fora do escopo: `CANCELLED`/`POSTPONED` tratados como `scheduled` por enquanto — decisão diferida (ver spec).

**Pré-requisito:** Planos 2.0, 2.1, 2.2, 2.3 concluídos.

**Critério de aceitação:**
- 4 unit tests do mapper de status (todos os valores do enum).
- Teste funcional do endpoint cobrindo: SCHEDULED → FINISHED com placar.
- Typecheck e suite passam.

```ts
import type { FootballDataMatchStatus } from '#integrations/football_data/types'
import type { MatchStatus } from '@palpites/shared'

export function mapMatchStatus(s: FootballDataMatchStatus): MatchStatus {
  switch (s) {
    case 'IN_PLAY':
    case 'PAUSED':
    case 'SUSPENDED':
      return 'live'
    case 'FINISHED':
    case 'AWARDED':
      return 'finished'
    case 'SCHEDULED':
    case 'TIMED':
    case 'POSTPONED':
    case 'CANCELLED':
    default:
      return 'scheduled'
  }
}
```

## Task 2: `RefreshMatchService`

**Files:**
- Create: `apps/api/app/services/refresh_match_service.ts`

- [ ] **Step 1: Implementar**

```ts
import { inject } from '@adonisjs/core'
import { DateTime } from 'luxon'
import FootballDataClient from '#integrations/football_data/client'
import MatchRepository from '#repositories/match_repository'
import { mapMatchStatus } from '#services/match_status_mapper'

export interface RefreshReport {
  matchId: string
  updated: boolean
  reason?: string
}

@inject()
export default class RefreshMatchService {
  constructor(
    private client: FootballDataClient,
    private matchRepository: MatchRepository
  ) {}

  async refresh(matchId: string): Promise<RefreshReport> {
    const match = await this.matchRepository.findByIdOrFail(matchId)
    const data = await this.client.fetchMatchById(match.externalId)
    if (!data) {
      return { matchId, updated: false, reason: 'match não encontrado no provider' }
    }

    await this.matchRepository.update(match, {
      homeScore: data.score.fullTime.home,
      awayScore: data.score.fullTime.away,
      status: mapMatchStatus(data.status),
      kickoffAt: DateTime.fromISO(data.utcDate),
    } as any)

    return { matchId, updated: true }
  }
}
```

## Resumo do Plano 2.4

- Refresh manual via `POST /matches/:id/refresh` (base pro scheduler do plano 3).
- Mapper de status cobre todas as variações do provider.
- `CANCELLED`/`POSTPONED` viram `scheduled` (simplificação consciente — iteração futura trata).

**Decisões registradas:**
- `kickoffAt` é atualizado no refresh (provider pode ajustar horário da partida).
- Se provider retornar `null` (match deletado externamente), **não apaga** local — só reporta; admin decide.
- `externalId` é o identificador do football-data (salvo no plano 2.3).
- Sem `MatchFactory` por ora — criação inline nos testes é suficiente; fábrica vem quando houver 3+ usos.
