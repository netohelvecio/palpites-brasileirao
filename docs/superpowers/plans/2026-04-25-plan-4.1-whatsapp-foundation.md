# Plano 4.1 — WhatsApp Foundation (env, client port, stubs, status endpoint)

> **For agentic workers:** REQUIRED SUB-SKILL: superpowers:executing-plans + superpowers:test-driven-development.

**Goal:** Estabelecer o esqueleto do módulo WhatsApp:
- Env vars (`WHATSAPP_MODE`, `WHATSAPP_GROUP_JID`, `WHATSAPP_AUTH_PATH`).
- Abstract class `WhatsAppClient` (port).
- Implementações `StubClient` (loga) e `DisabledClient` (sempre offline).
- Provider que registra a impl certa no container conforme `WHATSAPP_MODE`.
- `GET /whatsapp/status` real (substitui o placeholder do `HealthController`).
- `FakeWhatsAppClient` em `tests/helpers/`.

**Architecture:**
- Abstract class como token de DI (interfaces TS são apagadas, então usamos `abstract class`).
- Provider em `providers/whatsapp_provider.ts` registra o singleton no `register()`, escolhendo a impl pela env. Modo `real` ainda **lança "not implemented"** — `BaileysClient` chega no plano 4.5.
- Status endpoint sem auth (admin precisa ver mesmo se token mudou). Lê `client.mode` + `client.isConnected()`.

**Pré-requisito:** Spec `docs/superpowers/specs/2026-04-25-whatsapp-outbound-design.md`.

**Critério de aceitação:**
- `WHATSAPP_MODE=disabled` no `.env.test` → `app.container.make(WhatsAppClient)` retorna `DisabledClient`.
- `GET /whatsapp/status` retorna `{ status: 'ok', mode, connected }` (status:200) sem auth.
- `FakeWhatsAppClient` permite swap em testes (mesma forma do `FootballDataClient`).
- Typecheck + suite passa.

## Task 4: `WhatsAppProvider` + registro em `adonisrc.ts`

**Files:**
- Create: `apps/api/providers/whatsapp_provider.ts`
- Modify: `apps/api/adonisrc.ts`

- [ ] **Step 1: Provider**

```ts
import type { ApplicationService } from '@adonisjs/core/types'
import env from '#start/env'
import WhatsAppClient from '#integrations/whatsapp/whatsapp_client'

export default class WhatsAppProvider {
  constructor(private app: ApplicationService) {}

  register() {
    this.app.container.singleton(WhatsAppClient, async () => {
      const mode = env.get('WHATSAPP_MODE')

      if (mode === 'stub') {
        const { default: StubClient } = await import('#integrations/whatsapp/stub_client')
        return new StubClient()
      }

      if (mode === 'real') {
        // BaileysClient chega no plano 4.5. Por enquanto, lança.
        throw new Error(
          'WHATSAPP_MODE=real ainda não suportado — implementação chega no plano 4.5'
        )
      }

      const { default: DisabledClient } = await import('#integrations/whatsapp/disabled_client')
      return new DisabledClient()
    })
  }
}
```

- [ ] **Step 1: Criar helper**

```ts
import WhatsAppClient, { type WhatsAppMode } from '#integrations/whatsapp/whatsapp_client'

export class FakeWhatsAppClient extends WhatsAppClient {
  mode: WhatsAppMode = 'stub'
  private connected = true
  public sentMessages: string[] = []
  public throwOnSend: Error | null = null

  async connect() {
    this.connected = true
  }

  async disconnect() {
    this.connected = false
  }

  isConnected(): boolean {
    return this.connected
  }

  setConnected(value: boolean) {
    this.connected = value
  }

  async sendToGroup(text: string): Promise<void> {
    if (this.throwOnSend) throw this.throwOnSend
    this.sentMessages.push(text)
  }
}
```

`mode` é mutável aqui (não `readonly`) só pra que testes do status endpoint possam simular qualquer modo sem trocar env.

- [ ] **Step 2: Criar functional spec — RED**

```ts
import { test } from '@japa/runner'
import app from '@adonisjs/core/services/app'
import WhatsAppClient from '#integrations/whatsapp/whatsapp_client'
import { FakeWhatsAppClient } from '#tests/helpers/whatsapp_mock'

test.group('GET /whatsapp/status', () => {
  test('retorna mode e connected do client', async ({ client, assert }) => {
    const fake = new FakeWhatsAppClient()
    fake.mode = 'stub'
    fake.setConnected(true)

    app.container.swap(WhatsAppClient, () => fake)
    try {
      const res = await client.get('/whatsapp/status')
      res.assertStatus(200)
      assert.deepEqual(res.body(), { mode: 'stub', connected: true })
    } finally {
      app.container.restore(WhatsAppClient)
    }
  })

  test('reflete connected=false quando offline', async ({ client, assert }) => {
    const fake = new FakeWhatsAppClient()
    fake.mode = 'real'
    fake.setConnected(false)

    app.container.swap(WhatsAppClient, () => fake)
    try {
      const res = await client.get('/whatsapp/status')
      res.assertStatus(200)
      assert.deepEqual(res.body(), { mode: 'real', connected: false })
    } finally {
      app.container.restore(WhatsAppClient)
    }
  })

  test('endpoint é público (sem Authorization)', async ({ client }) => {
    const res = await client.get('/whatsapp/status')
    res.assertStatus(200) // não 401
  })
})
```

## Resumo do Plano 4.1

- Env vars validadas em `start/env.ts`; default seguro `disabled` em `.env.example` e `.env.test`.
- `WhatsAppClient` abstract class em `app/integrations/whatsapp/`.
- `StubClient` (logger) + `DisabledClient` (offline + throw em send).
- `WhatsAppProvider` registra singleton no container conforme `WHATSAPP_MODE`. Modo `real` lança "not implemented" — vem no 4.5.
- `GET /whatsapp/status` real, sem auth, lê `mode` + `isConnected()` do client.
- `FakeWhatsAppClient` para swap em testes.

**Decisões:**
- Abstract class em vez de interface: necessário para DI runtime do Adonis.
- `mode` como property no client (não em env): testes não precisam mexer em env pra cobrir todos os modos.
- `DisabledClient.sendToGroup` lança (em vez de no-op silencioso): se for chamado, é bug — gate deveria ter barrado.
